> **Repo**: Run from the openroad-demo root.

Review demo project changes for structure and consistency with existing demo projects.

**Always run `/review` first** for generic checks (tone, language, hacks, consistency), then continue with the demo-specific checks below.

## 1. Gather demo context

Read the README.md and BUILD.bazel for every **existing** project on the main branch (these are the reference style — serv, vlsiffra, etc.). Read defs.bzl for global conventions. Then read the current (working tree) versions of all **changed** project files.

## 2. Check project README style

Each project README must follow this structure (compare against serv and vlsiffra):

1. **Title line**: `# <Name> — <Short Description>`
2. **Intro paragraph**: 2-3 sentences. Links to upstream repo. Credits the author/org. Mentions the HDL.
3. **"What This Demo Builds"** section: module name in bold, bullet list with Architecture, Target frequency, Pipeline/ISA details.
4. **"Reported vs. Actual Results"** section: table with columns `Metric | Reported | Actual`. Metrics: Frequency, Cells, Area (μm²), WNS (ps), Power (mW). Brief context sentence above the table.
5. **Routed image**: `![<name> routed](../docs/<project>/route.webp)`
6. **"Future Improvements"** section: bullet list of concrete next steps with specific ORFS variable names.
7. **"Build"** section: code block with `bazel build` commands for _synth, _route, _final.
8. **"References"** section: upstream GitHub, papers, relevant issues/PRs.

Flag any missing sections, wrong section order, or style deviations. Projects that are incomplete (status != Done) may have extra sections like "Status" or "Continue from" — that's fine, but the core sections above should still be present.

## 3. Check BUILD.bazel style

Each project BUILD.bazel should:

- Load from `//:defs.bzl` using `demo_flow` and `demo_gallery_image`
- Use `demo_flow()` with explicit `arguments` dict (not relying on hidden defaults)
- Include organized comment headers: `# --- Synthesis ---`, `# --- Floorplan & Placement ---`, `# --- Speed defaults ---`
- Have a `demo_gallery_image()` pointing to `:<module>_route` (not an intermediate stage)
- Use `sources = {"SDC_FILE": [":constraints.sdc"]}` for constraints

Flag if:
- Gallery image points to anything other than `_route`
- Gallery image description doesn't indicate the stage for incomplete projects
  (e.g. "(GRT)"). Only add the stage label if it's NOT routed — routed is the
  default and doesn't need annotation
- Speed defaults are missing comments
- Arguments that should be in `_GLOBAL_SETTINGS` (defs.bzl) are duplicated per-project
- Chisel/generator steps are not clearly commented with step numbers

## 4. Check top-level README.md

- **Projects table**: new project row must have all columns filled. Status should accurately reflect how far the flow got (Done, CTS, Routing, etc.)
- **Gallery grid**: new project must appear with image link and description
- **Upcoming table**: project must be removed from Upcoming if it now has a directory
- **Column order**: Project, Description, PDK, Reported Freq, Reproduced Freq, Cells, Area, Version, Project version, Last updated, Status

## 5. Check constraints.sdc

- Must exist in the project directory
- Should set a realistic clock period for the target frequency
- Compare against serv/vlsiffra patterns for style

## 6. Check build_times.yaml

- If the project has completed stages, it should have an entry in build_times.yaml
- Entry should be generated by `bazelisk run //scripts:build_times -- <project>`, not hand-written

## 7. Check MODULE.bazel

- New project source must be fetched via `http_archive` with:
  - A pinned commit hash URL (not a branch)
  - `sha256` checksum
  - `strip_prefix` if needed
  - Patches listed in `patches` with `patch_args = ["-p1"]`
- No `git_repository` rules (avoids system git dependency)

## 8. Check collected logs/metrics/reports size

If the project has `logs/`, `metrics/`, or `reports/` directories, check their
total size. Run `du -sh <project>/logs/ <project>/metrics/ <project>/reports/`
and check individual file sizes with `du -h <project>/logs/* | sort -rh`.

- **Total < 1 MB**: fine, commit as-is
- **Total 1-5 MB**: acceptable but flag any individual file > 500 KB
- **Total > 5 MB**: too large. Suggest excluding the biggest log files from
  git and replacing them with a summary in the project README. For example,
  placement logs with verbose iteration output can be summarized as "3_3_place_gp:
  611s, overflow converged in 312 iterations" rather than committing the full log.

The goal is to keep collected data small enough for Claude to read directly.
Large binary-like logs (thousands of placement iterations, routing progress)
should be summarized, not committed verbatim.

## 9. Check gallery grid size

If the gallery grid in the top-level README has more than 4-5 columns, it will
be too wide for GitHub rendering. When this happens, suggest restructuring as a
2-row grid or reducing thumbnail sizes via the `thumbnail_size` parameter in
`demo_gallery_image()` (default 400px — try 300 or 250 for larger galleries).

## 10. Summarize demo-specific findings

Present findings as a numbered list of **concrete suggestions**, grouped by file. For each suggestion:

- State what's wrong or inconsistent
- Show what the reference projects do
- Give the specific fix (exact text or command to run)

Prioritize: policy violations > missing content > style nits.
