> **Repo**: Run from your ORFS workspace root (the one with `MODULE.bazel` and the `bazel-orfs` dep). Examples use the neutral `//<project>:<module>_<stage>` naming — substitute your own targets.

Diagnose and iterate on an OpenROAD/ORFS failure — a crash, a hang, a parallel race, or a nondeterministic result — and set up a fast edit/measure loop before you package a reproducer.

ARGUMENTS: $ARGUMENTS

This is the diagnosis layer that comes *before* packaging. Once you know
what the failure is and have a minimal `.tcl` that triggers it, hand off to:

- `/openroad-issue` — file it upstream as a `git am` patch + failing bazel test.
- `/untar-and-run-report` — ship it as an untar-and-run `.tar.gz` archive.
- `/odb-to-cpp` — turn a whittled `.odb` into a self-contained C++ unit test.

## 1. Decode the exit code — openroad does not terminate itself

openroad runs to completion or crashes in-process; it never sets its own
wall clock. So the exit code tells you *who* stopped it:

- **143 = 128 + SIGTERM(15)** → an *external* supervisor killed it. It hung
  or ran past someone's timeout.
- **137 = 128 + SIGKILL(9)** → usually OOM (the kernel) or a hard kill.
- **134 = SIGABRT**, **139 = SIGSEGV(11)** → an in-process crash/assert; the
  stack trace is in the log.

For a 143, find *who* sent it before assuming a tool bug. Check, in order:
the bazel test timeout (build actions have **no** bazel timeout — only tests
do), any `timeout`/`kill` wrapper in the invoking script, the ORFS
`flow/Makefile` `run` target (no timeout there by default), and finally the
CI runner's own container wall/idle watchdog. If nothing you can see fired,
the harness owns the timeout — which means **your reproducer must supply its
own** (a `timeout` in `run.sh`, §6).

## 2. Characterize a hang before calling it a deadlock

Sample the *real* openroad process, not the `make`/`tee`/`bash -c` parent
(match on `comm`):

```bash
pid=$(ps -eo pid,comm,args | awk '$2=="openroad"&&/repro.tcl/{print $1}')
top -bn1 -p $pid
ps -L -o stat= -p $pid | sort | uniq -c
ps -L -o wchan:30= -p $pid | sort | uniq -c
```

- **~0% CPU + all threads in `futex_*`** = a clean deadlock.
- **High CPU + mixed running / `futex_do_wait`** = a threaded livelock /
  lock-contention storm (burns cores, makes no progress).

**Race vs cost vs deterministic loop — the `-threads 1` test.** Re-run the
*same* input single-threaded:

- Completes fast at `-threads 1` but hangs at `-threads N` → **parallel data
  race / livelock**. This is the cleanest "it's concurrency, not cost"
  evidence. There is then *no* deterministic per-cut signal, so whittle won't
  converge (§6) — ship the full-size reproducer.
- Hangs even at `-threads 1` → deterministic infinite loop → whittle *can*
  shrink it cleanly.
- A *smaller* design hangs while a *bigger* one completes → the cost is
  anomalous → genuine bug, not just "big design is slow".

**gdb caveat (Heisenbug).** `ptrace_scope=1` blocks attaching to a
non-child, so launch openroad *under* gdb (`gdb --args openroad …`) to make
it a child. But ptrace serialization often *masks* a race (it completes under
gdb). Use `perf record --call-graph dwarf -p <pid>` on the live hang instead
— it doesn't perturb scheduling and usually names the hot function — and
point the maintainer at TSAN / `rr`.

## 3. Stage the exact environment with `_deps`

Every ORFS stage target has a companion `{stage}_deps` `pkg_tar`
(`include_runfiles = True`) that bundles the pinned openroad binary, the PDK,
generated macro LEF/LIB, the input ODB, the SDC, the ORFS `flow/scripts/`
(`open.tcl`, `setRC.tcl`), the config, and a `make` wrapper. Never
hand-assemble libraries — they drift.

```bash
bazelisk build //<project>:<module>_<stage> //<project>:<module>_<stage>_deps
```

The stage's *own* output ODB (e.g. `5_1_grt.odb`) is generated by the stage,
not shipped in its `_deps` (which carries the *previous* stage's ODB as
input). Get the current stage's ODB from the materialized output in
`bazel-bin/…/results/…/` (build without `--remote_download_minimal`) or by
running the stage's `make do-<substep>` from the extracted `_deps` tree.

**Untar gotchas (raw `_deps` tar, not the `bazel run` deploy):** the tar
ships the runfiles but not the deploy wrapper's `_main/external/<repo>`
symlinks or `_main/config.mk`. Recreate them so the ORFS make wrapper
resolves `external/orfs+/flow` and `DESIGN_CONFIG=config.mk` relative to
`_main`:

```bash
cd <stage>.runfiles
mkdir -p _main/external
for d in */; do r=${d%/}; [ "$r" = _main ] || ln -sfn "../../$r" "_main/external/$r"; done
cp -fL <top>/*.short.mk _main/config.mk
```

Run from `_main`. `OPENROAD_CMD = openroad -exit -no_init -threads $(NUM_CORES)`,
so `make run RUN_SCRIPT=repro.tcl NUM_CORES=<N> ODB_FILE=.../<stage>.odb`
reproduces the flow command exactly.

**No local cache hits across exec platforms:** CI runs in a container, so
even checking out CI's exact commit re-executes stages locally (the
remote-cache key includes platform properties). Plan for a full stage rerun
when the cache is cold.

## 4. Fast local iteration on an OpenROAD *source* change (`_deps` + bring-your-own binary)

When you are **modifying OpenROAD itself** (a patch to `rsz`/`grt`/…) and
want its effect on a real stage, do **not** iterate through the flow: the
openroad binary is a *tool input to every ORFS stage action*, so any edit to
it invalidates synth→floorplan→place→cts→… and forces a **full P&R rebuild
per edit** (hours — this is the trap). Decouple the binary from the frozen
prior stages via `_deps`:

1. **Freeze the inputs once, from a cached build.** In a checkout where the
   upstream stages are already in the remote cache, build only the stage's
   deps tar:
   ```bash
   bazelisk build //<project>:<module>_<stage>_deps   # cache hit → minutes
   ```
   It bundles the frozen previous-stage ODB + SDC + libs + `flow/scripts` +
   the *unpatched* pinned openroad + a `make` wrapper (§3). Built with the
   unpatched binary, so it stays a cache hit and your change never touches it.
2. **Build your modified openroad separately** (`git apply` your patch in the
   OpenROAD checkout, or carry it in `MODULE.bazel`):
   ```bash
   bazelisk build @openroad//:openroad   # only the changed module recompiles → fast incremental
   ```
3. **Run just that stage against the frozen deps, with your binary.** Extract
   the tar, recreate the `_main/external` symlinks + `config.mk` (§3 untar
   gotchas), then from `_main` run the ORFS make step. The extracted make
   wrapper **`export`s `OPENROAD_EXE` unconditionally**, so an `OPENROAD_EXE`
   *env var* is clobbered — pass it as a **make argument** instead (a
   command-line make variable beats both the env and the makefile default).
   Point at the patched binary **at its build location** (it needs the
   adjacent `.runfiles`; don't copy just the ELF):
   ```bash
   PATCHED=$(readlink -f .../bazel-bin/external/openroad+/openroad)
   cd <stage>.runfiles/_main
   sh ../../make_<...>_<stage> do-<substep> OPENROAD_EXE="$PATCHED" NUM_CORES=<N>
   ```
   Every prior stage is reused from the frozen tar, so each iteration is just
   the one stage (minutes), not a full flow.

Same `_deps` mechanism as §3, opposite direction: there you ship
binary-agnostic inputs *for a maintainer*; here you hold the inputs fixed and
swap the binary to *develop* an OpenROAD change (algorithm / effort knobs /
DSE).

## 5. Split a stage `.tcl` at an ODB checkpoint — re-run only a *read-only* late step

Within one stage the `.tcl` is a sequence (`load → route → estimate_parasitics
→ step → write`). For a **late** step that only *reads* the ODB + parasitics
— an STA/timing report, `report_clock_skew`, or a *placement*-stage
`repair_timing` (`estimate_parasitics -placement`, no router) — don't re-run
the whole stage. Checkpoint the ODB+SDC before the step, then load and run
only that step (seconds per run):

1. **Produce the checkpoint once — save BOTH `.odb` and `.sdc`.** Drop
   `orfs_write_db <cp>.odb` **and** `orfs_write_sdc <cp>.sdc` before the step.
   The SDC carries state the ODB doesn't (propagated-clock, the updated clock
   period from `write_ref_sdc`).
2. **Iterate with a tiny debug tcl** via `make run` (§4 deps tree) so
   liberty/PDK load exactly like the stage:
   ```tcl
   source $::env(SCRIPTS_DIR)/open.tcl   ;# reads $::env(ODB_FILE) + its .sdc + all liberty
   estimate_parasitics -placement        ;# re-derive the RC the step reads
   report_worst_slack -max               ;# the one read-only thing under test
   ```
   ```bash
   cd <stage>.runfiles/_main
   ODB_FILE=$(pwd)/…/<cp>.odb \
     sh ../../make_<...>_<stage> run RUN_SCRIPT=/abs/debug.tcl OPENROAD_EXE="$PATCHED" NUM_CORES=<N>
   ```
   `ODB_FILE` is a shell **env** var (open.tcl reads it); `OPENROAD_EXE` stays
   a make **arg** (§4).

**Caveat — this does NOT work for `repair_timing` *after* global route.**
repair-after-GRT mutates the netlist and drives *incremental* `global_route`
as it buffers/clones, so it needs the live in-session `GlobalRouter`, which
the ODB round-trip loses: a bare reload fails with `RSZ-0074` ("failed to
build tree from global routes"); adding `read_guides <RESULTS_DIR>/route.guide`
gets past that but then hits `GRT-0265` ("segment not H/V/via") during the
incremental parasitic update. For grt-repair, iterate via §4 (full grt stage
on the frozen deps + BYO binary — the route re-runs per edit, but
place/cts/synth don't). Rule of thumb: §5 for steps that *read* the ODB; §4
for steps that *re-drive* the router.

## 6. Reproducer shape: self-contained tcl + bring-your-own-openroad

Do **not** make the reproducer drive the `make` wrapper or depend on exported
env vars — the maintainer won't have that, and they want to run it under a
debugger. Instead:

- **`repro.tcl` locates its own inputs**, so `openroad repro.tcl` works from
  anywhere with no setup:
  ```tcl
  set RF [lindex [glob [file dirname [file normalize [info script]]]/make_*_<stage>.runfiles] 0]
  sta::set_thread_count 32   ;# the one knob; edit to change (some bugs need many threads)
  read_liberty $RF/.../*.lib ; read_db -hier $RF/.../<prev>.odb ; read_sdc $RF/.../*.sdc
  source $RF/.../setRC.tcl ; estimate_parasitics -placement ; sta::find_timing
  ```
  List the `read_liberty` lines explicitly (the exact set from the failing
  log) — don't glob a whole NLDM dir or you'll pull duplicate date-stamped
  corners. Bundle the staged `.runfiles` (libs, ODB, SDC, setRC) from the
  `_deps` tree; the maintainer supplies the binary.
- **`run.sh` is short and readable** — takes the openroad path, loops,
  reports the first hang (this is where the reproducer supplies the timeout
  the harness used to own, §1):
  ```bash
  #!/usr/bin/env bash
  set -u
  openroad=${1:?usage: ./run.sh /path/to/openroad}
  [ -x "$openroad" ] || { echo "not executable: $openroad"; exit 2; }
  cd "$(dirname "$0")"; mkdir -p logs
  for i in $(seq 1 10); do
    echo "run $i ..."
    timeout 420 "$openroad" -exit -no_init -no_splash repro.tcl >"logs/run_$i.log" 2>&1
    [ $? -eq 124 ] && { echo "HANG on run $i (log: logs/run_$i.log)"; exit 0; }
  done
  echo "no hang in 10 runs — intermittent, run again"; exit 1
  ```
- `chmod +x run.sh` **before** packaging. Tarball as `.tar.gz`.
- **Do NOT ship the openroad binary in a bring-your-own reproducer.** A
  `_deps` tree ships the pinned openroad (§3, for local staging), so when you
  package a `_deps`-based bundle **strip `openroad+/openroad` (and `-qt`)
  first**: it's hundreds of MB of dead weight and muddies which binary is
  under test. Make `run.sh` take the openroad path as a *required* arg — no
  bundled default. (When the bundle drives the ORFS `make` wrapper, pass it as
  the `OPENROAD_EXE=<path>` **make argument** — §4 — not an env var.)
- Include a `diagnostics/` dir (perf data + a short writeup) so the deep
  evidence travels with the tarball, keeping the issue body lean.

## 7. Minimize with whittle.py — carefully for intermittent bugs

`whittle.py` (in the OpenROAD tree, `etc/whittle.py`) delta-debugs an `.odb`
by cutting instances then nets, re-running `--step` and keeping cuts where
`--error_string` still appears in the output. See `/openroad-issue` step 4
for the standard flow and flags. The nuances that matter for a *hang* or a
*race*:

- **A timeout does NOT count as the error** (whittle just breaks and returns
  "not found"). For a hang, the step must turn the hang into the string
  itself — run openroad under an internal `timeout` and `echo` a sentinel on
  exit 124; pass `--error_string SENTINEL --use_stdout`:
  ```bash
  # whittle_step.sh — emit sentinel when openroad hangs
  timeout 180 "$OPENROAD_EXE" -no_init -threads 32 -exit repro_whittle.tcl 2>&1
  [ $? -eq 124 ] && echo "WHITTLE_HANG_DETECTED"
  ```
  ```bash
  python3 etc/whittle.py --base_db_path <prev>.odb --error_string WHITTLE_HANG_DETECTED \
    --step ./whittle_step.sh --persistence 1 --multiplier 1 --use_stdout --timeout 700
  ```
- **A missed reproduction only makes whittle keep *more* of the design** (the
  safe direction).
- **A pure parallel race usually will not whittle at all.** If the `-threads 1`
  test (§2) showed a livelock/race (single-threaded completes, only the
  threaded run hangs), whittle has no deterministic per-cut signal: cuts that
  keep the bug fail to hang within the repeat budget and get reverted, so it
  oscillates and never reduces. Don't burn hours on it. Ship the full-size
  ODB reproducer — for a flaky bug a large *reliable* repro beats a tiny one
  that fires 1-in-20 — and let the `-threads 1`-vs-`N` contrast + perf hotspot
  + a TSAN pointer do the work in the issue. Whittle only when the failure is
  *deterministic* (crash/assert/`-threads 1` hang with a stable
  `--error_string`): then use coarse settings and stop the moment the hit rate
  drops.

## 8. OpenSTA-from-netlist may not reproduce an ODB-backed bug

A pure-OpenSTA repro (`opensta` + `read_verilog netlist.v` + `link_design` +
`read_sdc` + `find_timing`, built via `bazelisk build @openroad//src/sta:opensta`)
is the most portable *when it works* — but `write_verilog`→`read_verilog` can
normalize away the trigger (e.g. constant/tie nets that drive `sta::Sim` /
`sta::Bdd` constant propagation). Verify it actually reproduces before
shipping it; if it doesn't while the `read_db -hier` ODB path does, that's a
useful finding (the bug needs the dbSta/ODB representation) — keep the
ODB-based reproducer.
